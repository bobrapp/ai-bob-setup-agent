First run

Before you read further, you should have completed the install guide. That means make doctor exits clean and your .env has at minimum ORGO_API_KEY, OPENAI_API_KEY, TELEGRAM_BOT_TOKEN, and TELEGRAM_CONTROL_CHAT_ID.

Verify the doctor passes:

$ make doctor

The doctor prints a table of required and optional environment keys. Required keys missing → red exit code 1. Fix those before running anything that touches production.

Take a tour of the help:

$ python -m src.setup_agent --help
$ make help

Daily workflows

Three commands cover ~90% of operator time:

GoalCommand
Onboard a new customermake onboard CUSTOMER=<slug>
Add an agent to an existing customermake add-agent CUSTOMER=<slug> AGENT=<name>
Check fleet healthmake health
Run the watchdog loopmake watchdog
Decommission a customermake decom CUSTOMER=<slug>
List configured customerspython -m src.setup_agent list

Every command takes an optional DRY_RUN=true to walk the orchestration path without making real API calls. Useful before a risky change or when you want to audit what would happen.

Onboarding a new customer

1. Discovery call

Out of scope for this tool — but the output is the customer config. Decide the tier (Hermes or OpenClaw), the agents you'll provision (1–3 typically), and the MCPs and connectors each agent needs.

2. Write the customer YAML

Copy config/customers.example.yaml to config/customers/<slug>.yaml and edit. The slug must be lowercase, hyphens-only — it's used in workspace names and URLs. Validate it parses: 

$ python -c "from src.config import load_customer; load_customer('acme-marketing')"

3. Dry run

$ make onboard CUSTOMER=acme-marketing DRY_RUN=true

Watch the JSON logs. You should see orgo.workspace.create, one orgo.cloud_computer.create per agent, a sequence of hermes.remote events for runtime bootstrap, MCP installs, connector installs, and a final onboard.done. No errors.

4. Real run

$ make onboard CUSTOMER=acme-marketing

Should complete in 2–5 minutes depending on how busy Orgo's provisioning queue is. You'll get a Telegram notification when finished. The structured log captures every step.

5. Hand off to the customer

Send the customer the Telegram channel invite plus a one-page operating cheatsheet — what each agent does, how to talk to them, how to escalate. Template lives at docs/customer-handoff-template.md (TODO: ship in v0.2).

Managing agents

Adding an agent mid-contract

Customer asks for a new agent. Edit the customer YAML to add the agent block, then: 

$ make add-agent CUSTOMER=acme-marketing AGENT=billing-reminder-agent

Removing an agent

Remove the agent block from the YAML, then manually decommission the cloud computer from the Orgo console (auto-remove is deliberately not wired — too easy to nuke production by accident). v0.2 will add a remove-agent command with a confirmation prompt.

Updating agent config

Edit the YAML, re-run onboard. Idempotent — workspace and cloud computers stay, MCP / connector changes apply, runtime gets reloaded.

Observability

One-shot health check

$ make health

Walks every customer in config/customers/ and lists each agent with its status. Exit code 0 if all healthy, 1 if any are degraded or down. Cron this for a poor-man's monitoring loop.

Long-running watchdog

$ make watchdog

Polls every WATCHDOG_INTERVAL_SECONDS (default 300). On a missed heartbeat, fires Telegram and email alerts. Run as a systemd unit or in a tmux session for sustained operation.

What does "down" mean?

  • healthy — cloud computer status is "running".
  • degraded — cloud computer status is "provisioning" (Orgo is still spinning it up).
  • down — status is "stopped" or "error", or the cloud computer is missing entirely.
  • unknown — Orgo returned a status we don't know how to interpret. Treat as warning.
✓
The watchdog is the moat.

Per the source episode, this is the difference between a hobby and a $1M business: catching agent-down before the customer notices. Treat watchdog tuning as core product work, not a side concern.

Telegram meta-agent

The Telegram channel is the operational interface. Outbound (alerts, notifications) is fully wired. Inbound (commands you send the bot) is currently a documented extension point — implement using python-telegram-bot's Application + CommandHandler per the docstring in src/telegram_meta.py.

Suggested command set

CommandBehavior
/statusList customers and agent health.
/restart <cust> <agent>Restart a specific agent.
/install <cust> <agent>Add a new agent for an existing customer.
/logs <cust> <agent>Tail the agent's recent operational logs.
/panicPause all provisioning, page the operator immediately.

Troubleshooting

"OrgoError: ORGO_API_KEY missing"

Fill in ORGO_API_KEY in .env, or run with DRY_RUN=true if you're just walking the orchestration.

"Customer config validation failed"

Pydantic prints the exact field. Most common: slug not lowercase / not alphanumeric-hyphen, tier value not hermes or openclaw, MCP name not in perplexity / context7 / x.

"Telegram notifications not arriving"

Verify TELEGRAM_BOT_TOKEN is set, the bot is added to the chat, and TELEGRAM_CONTROL_CHAT_ID matches the chat's actual ID. The notifier fails quietly by design — check structured logs for telegram.send_failed.

"Watchdog reports everything as 'unknown'"

Orgo returned a status string the client doesn't recognize. Open an issue with a sample log line and we'll add the mapping in src/observability.py → _evaluate.

"Re-running onboard fails because workspace exists"

It shouldn't — the client is idempotent. If it does, check that the customer slug in the YAML matches the workspace's stored slug. Slug mismatches break the idempotency lookup.

"CI dry-run onboard fails but local works"

Most likely: config/customers/ isn't in the CI workspace because it's gitignored. The CI workflow copies the example YAML into place — verify your test customer YAML isn't relying on local-only state.

FAQ

Can I run this without Orgo?

Not today. The cloud-computer abstraction is mapped to Orgo's API surface. Swapping platforms is a focused refactor on src/orgo_client.py; the rest of the code is platform-agnostic.

Can I run multiple operators against the same repo?

Yes — the repo is just code and YAML. Each operator should have their own .env with their own Orgo key. Don't try to share .env via git; that's how secrets leak.

How big can a single operator scale this?

Nick's reference deployment manages 27 cloud computers across many customers from one Telegram channel. With the watchdog working correctly, we don't see a structural ceiling below about 50 cloud computers per operator. Beyond that, hire.

Where do per-customer agent prompts live?

Not in this repo. The customer's actual agent logic — prompts, tool selection, business rules — is written separately and committed to the customer's workspace via the Hermes /OpenClaw runtime. This repo handles provisioning and observability; what the agents actually do is the customer-specific work Bob does between onboardings.

How do I contribute?

Open a PR. The CI gate is: ruff check clean, ruff format --check clean, all smoke tests passing, dry-run onboard passing. Anything beyond that is up for discussion.
