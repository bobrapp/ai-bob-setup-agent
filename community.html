The four jobs to be done

What we actually need agents to do.

Before evaluating tools, name the work. Every option below is graded against how cleanly it can ship these four roles — not against generic "AI features" marketing.

🤝 Welcomer

Every new member gets a personalized DM and a welcome-thread post within 5 minutes of joining. Pulls one thing from their intro to make it feel human, not boilerplate.

real-timeper-member

📰 Curator

Weekly digest of the best conversations. Drafts member-spotlight posts. Surfaces stale threads that deserve a bump. Asks Bob / Ken to approve before publish.

weeklyhuman-approved

🛡️ Moderator

Scans every post and comment for spam, toxicity, PII, and scam links. Flags to a human review queue. Never auto-removes in Phase 3 — human in loop is required.

📜 Auditor

Sequential log of every action — who did what, when, with which model, why. Append-only to docs/build-log.md + JSONL. Non-negotiable per the AIGovOps Foundation provenance rule.

Hard truths from the research

The gotchas you only find at the bottom of the docs.

!
Circle.so Business is $199/mo, not $299/mo — but webhooks need Circle Plus.

Outbound "Send to webhook" actions require Circle Plus (custom-priced via sales). On Business we poll the Admin API instead — fine for hundreds of members, tight at scale because the Business quota is 5,000 API requests per month. Plan to upgrade to Plus only when push events become necessary.

!
Circle has no moderation API surface.

The flag / approve / reject queue is admin-UI only. Our Moderator agent has to pull, classify, and act via the Admin API (which can delete posts and deactivate members). No push event fires when content gets flagged by Circle's built-in rules — so we ride alongside, not on top of, Circle's native moderation.

×
Marblism and Sintra both fail the AIGovOps audit test.

Neither exposes customer-accessible per-action logs with timestamps and inputs/outputs. Sintra has no API at all. Marblism has 24/7 platform logging but no documented export of agent actions to the customer. Both are wrong for an AIGovOps Foundation deliverable regardless of cost.

✓
The DM-via-API path exists but is fiddly.

The Admin API has no "send DM" endpoint. To send a Welcomer DM you mint a JWT via the Headless Auth API impersonating a designated bot member, create a direct ChatRoom, then post the message. Tokens expire every hour. Workable; just be honest about the boilerplate.

Recommendation

Ship the hybrid. Keep our governance. Pay $211/mo.

Of the seven, the hybrid path is the only one that gets all four jobs done at low cost without breaking the AIGovOps audit rule. Native Hermes is technically purest but spends two weeks of build time on undifferentiated plumbing. Marblism and Sintra fail the audit requirement on day one. n8n is excellent but adds devops we don't need to take on right now. Zapier gets expensive fast at community-management volume. Make.com alone is great but doesn't give us the AIGovOps provenance chain natively. Circle's native AI Workflows are free-in-plan but vendor-bound.

The hybrid composition:

  • Circle native AI Workflows handle the boring in-platform basics — onboarding flows, profanity filtering, tag-on-keyword. Free in our plan. No code.
  • Make.com Core ($12/mo) handles cross-platform orchestration — pulling intros into a draft, queueing the Welcomer DM, sending the weekly digest to Bob & Ken for approval before publish, running the AI Moderator classifier on a 5-minute poll.
  • A thin Hermes Auditor shim in src/community/audit.py subscribes to Make.com's webhook output and writes every action to the AIGovOps audit log (JSONL + docs/build-log.md) with model, prompt, response, and result. This is the piece that keeps us AIGovOps-compliant. ~150 lines of Python on the existing audit_log.py pattern.

Each piece is independently swappable. If Circle ships a deeper AI product, we delete Make scenarios. If Make.com prices break us, we move scenarios to n8n on a $5 droplet without changing the Auditor. We don't lock ourselves out.

✓
Why not Native Hermes only?

We'd ship it eventually and own everything — but we'd spend ~80% of the time on Circle API plumbing that Make.com already solves. The hybrid lets us put engineering effort where it differentiates AIGovOps (the audit shim, the moderation policy yaml) and lets a $12/mo tool handle the rest.

Open questions for Ken

Five things to settle before Phase 1.

  1. Circle plan tier today. Are we on Business ($199), Professional ($89, no API), or something else? If Professional, Phase 1 needs the upgrade as a pre-req.
  2. Bot member account. Welcomer DMs require a designated Circle member to impersonate. Create aigovops-welcomer@aigovops.community or reuse an existing seat?
  3. Moderation policy. Should the Moderator ever auto-remove (e.g., obvious scam links) or always flag-only on day one? Recommend flag-only for at least the first 4 weeks.
  4. Telegram review channel. Use the existing AIGovOps Telegram control chat (per telegram_meta.py) or stand up a new one specifically for community alerts? Volume separation matters.
  5. Webhook upgrade trigger. What's the latency tolerance that flips us from Business polling → Circle Plus webhooks? Recommend ≥1k active members or ≥1 moderation-relevant post per minute as the threshold.

Bottom line

Don't overbuild. Don't lock in.

We have a community to run and we have hours, not weeks. The hybrid path ships in days, costs ~$211/mo (most of which is Circle, which we're paying anyway), and keeps AIGovOps provenance native through the Auditor shim. Every component is swappable. Every action is logged. The agents do the work; Bob and Ken stay the humans in the loop where it matters.
