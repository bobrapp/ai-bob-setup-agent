Step 1 — Vertical Templates 612e10b

The project started with a question: whose problems are we actually solving? Nick Vasilescu's framework is clear — pursue the hungry, avoid the regulated. That meant building customer templates that matched the six verticals he identified: marketing agencies, law firms, insurance agencies, manufacturing, wholesale, and real estate. Three already existed. Three needed to be created.

This first commit added YAML configuration templates for manufacturing, wholesale, and real estate customers alongside the existing marketing, law, and insurance verticals. Each template captures the customer's agent stack (Hermes vs OpenClaw, how many of each), MCP requirements, second-brain seed file path, and notification preferences. The structure is intentionally declarative — no logic lives in the config, only intent.

Having all six verticals in place before writing the CLI meant the provisioning code could be tested against the full range of real-world shapes. A manufacturing customer with a single OpenClaw agent is structurally different from a marketing agency running two Hermes instances plus a dedicated outbound agent. Getting the config surface right first prevented expensive refactors later.

i
6 verticals, 1 schema

All templates share the same YAML schema. The provisioning engine reads a single config and routes to vertical-specific logic only where the stacks genuinely differ.

Step 2 — Production CLI 7906fcc

A config file is inert without something to execute it. This commit built the beating heart of the project: the make onboard CUSTOMER=acme command that reads a customer YAML and runs the full provisioning sequence end-to-end. The design goal was a single command that an operator could hand off — run this, watch the output, know when it's done and what it cost.

Pre-flight validation was the first concern. Rather than letting provisioning fail halfway through — leaving orphaned workspaces and half-configured VMs — a PreflightResult dataclass collects errors and warnings before a single API call is made. Errors are hard stops. Warnings proceed with a logged notice. This distinction matters operationally: a missing optional MCP config is a warning; a missing customer name is an error that should never reach production.

The CLI ships four commands: onboard, status, validate, and list. Rich terminal output uses progress panels so the operator can watch provisioning happen in real time rather than staring at a blinking cursor. Cost estimation is calculated and displayed before any resources are created — for acme-marketing (2 Hermes + 1 OpenClaw), that's $25K/mo, broken down by agent. The test suite grew from 15 to 33 tests with this commit, covering each CLI command and the pre-flight logic paths.

!
Cost estimation is a floor, not a ceiling

The displayed cost covers agent runtime only. Composio usage, Agent Mail volume, and Obsidian sync are additive. Build this into your customer contracts.

i
CLI commands

onboard — full provisioning run. status — check a customer's live environment. validate — dry-run the config without touching APIs. list — show all configured customers and their current state.

Step 3 — Systemd Watchdog 28240ee

Provisioning a customer environment is a one-time event. Keeping it running is a continuous obligation. The watchdog is the mechanism that converts a provisioned environment into a monitored service. This commit created the systemd unit file that runs the monitoring process on the operator's host machine, configured to restart automatically on failure and to emit heartbeats every minute.

Security hardening was non-negotiable. The unit file includes NoNewPrivileges=true, ProtectSystem=strict, and MemoryMax=256M. A watchdog process running with credentials for every customer's environment is a high-value target. Restricting it to the minimum required privilege surface and capping its memory footprint are basic hygiene for a production deployment.

The install and uninstall scripts both support a --dry-run flag, printing what they would do without touching the filesystem or systemd. An env file template ships alongside the unit file, documenting every variable the watchdog reads at startup. The combination makes onboarding a new operator machine a fifteen-minute task rather than a half-day archaeology exercise.

i
Designed for the operator's machine

The watchdog runs on the operator's host, not inside a customer workspace. This keeps the monitoring plane separate from the customer data plane — a clean architectural boundary that simplifies incident response.

Step 4 — Install Guide Update 5ab1d54

Software without documentation is a private tool. Software with documentation is a product. This commit added the "Deploy the watchdog service" section to the GitHub Pages install guide, turning the systemd work from step 3 into something an operator could actually follow.

The new section covers the complete watchdog setup flow: copying the unit file, populating the env file, running systemctl enable and systemctl start, and verifying the service is active. Management commands — reload, restart, view logs — are documented as explicit examples rather than left to the reader to derive. Two new troubleshooting entries address the failure modes most likely to appear in a first deployment: unit file not found (wrong path) and watchdog failing to start (env file misconfiguration).

The philosophy here is that the install guide is the product's first impression for every new operator. Every ambiguity in the guide becomes a support request. Writing the troubleshooting entries before anyone has filed a bug report is cheaper than responding to the same question ten times.

!
Keep the guide and the code in sync

The install guide was updated in the same commit as the watchdog code. Letting documentation lag creates a window where following the guide produces a broken installation.

Step 5 — Healthcheck Script 492f306

The watchdog monitors continuously; the healthcheck script answers a point-in-time question: right now, which customers are healthy and which are not? This commit built scripts/healthcheck.py, a standalone script that walks every configured customer, queries each agent's status endpoint, and presents the results in a rich terminal table with colored status icons.

Two output modes were built from the start. Human mode produces the colored table — readable at a glance, useful during an incident. Machine mode (--json) emits structured JSON suitable for ingestion by Datadog, PagerDuty, or any other monitoring tool the operator runs. The exit code contract is explicit: 0 means all customers healthy, 1 means issues detected, 2 means no customers configured. Monitoring integrations can key off the exit code without parsing output.

Single-customer filtering (--customer acme-marketing) was added for incident response workflows where you want to focus on one environment without noise from the rest. Dry-run support lets operators test the script's config parsing without making live API calls — useful when setting up a new machine before credentials are fully configured. Tests grew from 33 to 38 with this commit.

i
Exit code contract

Exit 0 = all healthy. Exit 1 = one or more customers reporting issues. Exit 2 = no customers found in config. Monitoring tools can act on exit code alone, no output parsing required.

Step 6 — Decommission Flow 8ac6e84

Every provisioned environment eventually needs to be torn down. A customer churns, a contract ends, a workspace needs to be rebuilt from scratch. The decommission flow is the inverse of onboarding — and it needs to be at least as reliable, because partial teardowns leave orphaned billable resources.

The core data structure is a DecomResult dataclass that tracks the outcome of each teardown step independently. The flow runs four stages: discover resources, confirm intent, delete, notify. The confirmation prompt is intentionally interactive — destructive operations require the operator to type the customer name to proceed. This is friction by design. Mistyping acme-marketing during a status check is a recoverable error. Mistyping it during a decommission is not.

Per-resource error handling ensures a failure in one step does not abort the rest. If a VM deletion fails — perhaps the VM was already terminated — the flow logs the error, marks that resource as failed in the DecomResult, and continues to the next resource. The final summary displays what was deleted, what failed, and the resulting monthly savings. Tests grew from 38 to 42.

!
Interactive confirmation is non-negotiable

The decommission flow will not proceed without an explicit typed confirmation. Scripted teardowns that bypass this prompt are unsupported. The friction is the feature.

i
Partial failure handling

A failed resource deletion does not abort the decommission. The flow continues, logs the error, and includes the failure in the final report. Manual cleanup is flagged explicitly rather than silently skipped.

Step 7 — Add-Agent Flow c14b1d0

Customers grow. A law firm that started with one Hermes intake agent adds a second for document drafting. A marketing agency expands from outbound to inbound. The add-agent flow handles this common lifecycle event without requiring a full decommission and re-provision of the customer environment.

The five-step provisioning flow validates the requested agent config first, listing all available agent types if the requested type is not recognized — a small UX detail that saves operators from hunting through documentation when they mistype a slug. Pre-flight checks run next, then workspace duplicate detection before any provisioning begins. If the agent already exists, the flow exits cleanly with a clear message rather than attempting to create a duplicate.

The summary step shows the cost delta explicitly: if the customer was at $15K/mo before and the new agent brings them to $25K/mo, the operator sees both numbers and the difference. This is useful context for the follow-up conversation with the customer about the invoice change. Tests grew from 42 to 46 with this commit, completing the core operational lifecycle: onboard, add-agent, healthcheck, decommission.

i
Cost delta display

The add-agent summary shows the before and after monthly cost side by side. Operators can communicate the pricing impact to customers before the next invoice cycle, not after.

Step 8 — AIGovOps Foundation Rebrand (this commit)

The final step is a change in identity more than in code. The project is being rebranded as an AIGovOps Foundation open source initiative — a signal that the work here is meant to be shared, studied, and built upon by any operator running AI agent businesses at scale. Provenance is governance: every change in an AIGovOps project carries the operator identity, the model that assisted, the timestamp, and the prompt that initiated the change.

Three additions ship with this commit: audit logging that records every provisioning action with full context, a configuration screen that surfaces the currently loaded customer configs and their health status, and this documentation page. The audit log is not an afterthought — it is the mechanism by which the AIGovOps Foundation's governance claims become verifiable. A system that says it logs everything means nothing without the infrastructure to prove it.

The documentation page exists because "getting to yes" is not just about shipping features — it is about giving the next operator a map. Every decision in this build session had a reason. Every architectural choice traded something. Writing that down is how institutional knowledge survives the context window.

A
AIGovOps Foundation

Open source. Operator-first. Every action traceable. The goal is a world where AI agent businesses are run with the same operational rigor as any other managed service.

!
Audit logs are immutable by design

The audit log backend uses append-only writes. Operators cannot edit or delete log entries. This is intentional — governance only works if the record cannot be quietly revised after the fact.

Eight commits. Forty-six tests. Six verticals covered. Four lifecycle operations shipped. One operator with a config file and a host machine can now provision, monitor, grow, and tear down customer environments without touching a cloud console. That's the lever.
