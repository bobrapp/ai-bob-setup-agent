Install Guide

Zero to first customer.
One afternoon.

Fresh laptop or VPS to a working ai-bob-setup-agent. Eight steps. Most of them take a minute; one of them is "wait for Orgo to give you a key."

~15 min hands-on if you already have API keys. ~2 hours if you need to apply for keys (the Orgo onboarding call is the longest blocker).

0. Prerequisites

You need:

  • Python 3.10+ — check with python3 --version.
  • Git — git --version.
  • Curl — likely already installed.
  • A GitHub account with a SSH key configured.
  • Mac, Linux, or WSL on Windows. (Pure Windows works but install.sh assumes bash.)

You should be acquiring:

  • Orgo API key — book Nick's 15-minute setup call at startup-ideas-pod.link/orgo_ai.
  • Hermes license — comes with Orgo enterprise tier.
  • OpenAI API key — for GPT 5.5 (the default model).
  • Telegram bot — talk to @BotFather in Telegram, run /newbot, save the token. Then start a chat with your new bot and grab the chat ID from https://api.telegram.org/bot<token>/getUpdates.

Optional but recommended:

  • Anthropic API key — for Opus 4.7 (long-horizon coding).
  • Z.AI key — for GLM 5.1 (cheap volume).
  • Kimi API key — for Kimmy (cheap light fallback, OpenClaw tier).
  • Perplexity API key — for the live-search MCP.
  • Context7 API key — for the live-docs MCP.
  • X API bearer token — for the social-signals MCP.
  • Composio API key — for app-action connectors (Gmail, Trello, Notion, etc.).
  • Agent Mail API key + domain — for per-agent email identities.
  • SMTP credentials — for email alerts. Any provider works; Postmark and Resend are popular.

1. Clone the repo

$ git clone https://github.com/bobrapp/ai-bob-setup-agent.git
$ cd ai-bob-setup-agent

2. Install

One command:

$ make install

Behind the scenes install.sh:

  1. Verifies Python ≥ 3.10.
  2. Creates a .venv if missing.
  3. Pip-installs everything in requirements.txt.
  4. Copies .env.example to .env if you don't have one.
  5. Runs the doctor.

The script is idempotent — safe to re-run. If you've already done steps 1–3 in some other way, pass flags: 

$ bash ./install.sh --skip-pip       # don't touch pip
$ bash ./install.sh --skip-deps      # don't check OS deps
$ bash ./install.sh --dry-run        # show what would happen

3. Fill in .env

Open .env in your editor:

$ $EDITOR .env

Fill in at minimum:

  • ORGO_API_KEY
  • OPENAI_API_KEY
  • TELEGRAM_BOT_TOKEN
  • TELEGRAM_CONTROL_CHAT_ID

Everything else is optional for the first run. You can come back and add MCPs, Anthropic, SMTP later — the system runs in a degraded-but-functional mode without them, with structured warnings telling you what's missing.

!
Never commit .env.

.env is gitignored. .env.example is the template that lives in version control. If you ever see a real key in a diff, you have a problem.

4. Run the doctor

$ make doctor

Prints a table of every required and optional environment key with its status. Tries an Orgo health-ping if the API key is set. Exit code 0 = ready, 1 = required keys missing.

5. Write your first customer config

Copy the example and edit:

$ mkdir -p config/customers
$ cp config/customers.example.yaml config/customers/acme-marketing.yaml
$ $EDITOR config/customers/acme-marketing.yaml

At minimum, change:

  • customer.slug — lowercase, hyphens only. Used in URLs and workspace names.
  • customer.legal_name and contact info.
  • customer.vertical — one of marketing, law, insurance, manufacturing, wholesale, real_estate.
  • contract.tier — hermes or openclaw.
  • agents — list of agents you'll provision. Most customers start with 1–2.

Validate the YAML parses cleanly:

$ python -c "from src.config import load_customer; print(load_customer('acme-marketing').customer.legal_name)"

6. Dry run the onboard

$ make onboard CUSTOMER=acme-marketing DRY_RUN=true

Walks every step of the orchestration without making real API calls. Returns a stub workspace ID, simulates cloud-computer creation per agent, simulates the Hermes install, MCP wiring, and connector installs. Telegram notification is logged but not sent.

Expected end of output: 

Onboarded acme-marketing (3 agents)
  • outreach-agent on hermes (3 MCPs, 2 connectors)
  • proposal-agent on hermes (2 MCPs, 1 connectors)
  • ops-agent on openclaw (1 MCPs, 1 connectors)

If you see errors at this stage, the customer YAML is the most likely culprit. Check the validation error message for the exact field.

7. Real run

!
This makes real API calls and creates real Orgo resources.

It will also bill against your Orgo account from this point forward. Make sure you mean it.

$ make onboard CUSTOMER=acme-marketing

Expect ~2–5 minutes. Watch the JSON logs:

{"event": "onboard.start", "customer": "acme-marketing", "agents": 3, ...}
{"event": "orgo.workspace.create", "customer_slug": "acme-marketing", ...}
{"event": "orgo.workspace.existing", "id": "ws_xxx", ...}
{"event": "orgo.cloud_computer.create", "agent_name": "outreach-agent", ...}
{"event": "hermes.install.start", "agent": "outreach-agent", ...}
{"event": "mcp.install", "mcp_name": "perplexity", ...}
{"event": "hermes.install.done", "agent": "outreach-agent"}
...
{"event": "onboard.done", "customer": "acme-marketing", "agent_count": 3}

You'll get a Telegram notification when finished.

Sanity check the fleet:

$ make health

All agents should show healthy. If any show degraded, give it 60 seconds — Orgo's still spinning up the cloud computer. Re-run.

8. Deploy the docs site (optional)

If you've forked this repo, GitHub Pages will auto-deploy the docs site on every push to main via the deploy-pages.yml workflow. Enable Pages once:

  1. Go to github.com/<your-user>/ai-bob-setup-agent/settings/pages.
  2. Source: GitHub Actions (not "Deploy from branch").
  3. Trigger the workflow once via the Actions tab or push a commit.
  4. Your site is at <your-user>.github.io/ai-bob-setup-agent.

To deploy from your laptop instead:

$ make deploy   # commits and pushes; Pages deploy runs automatically

9. Deploy the watchdog service

The watchdog is a lightweight background process that polls every customer's Orgo cloud computers on a configurable interval (default: 5 minutes). When a heartbeat is missed or a VM status changes to stopped or error, it fires alerts via Telegram and email.

Requires: Linux with systemd (Ubuntu 20.04+, Debian 11+, RHEL 8+). The watchdog is not needed during development — skip this section until you have production customers to monitor.

Install the systemd service

Make sure you've already run make install on the target machine so the .venv exists. Then: 

$ sudo bash ./deploy/install-watchdog.sh

The installer detects the repo owner automatically, templates the unit file with the correct paths, creates /etc/ai-bob-setup-agent/ for the env file, and enables the service to start on boot.

Use --dry-run to preview what the script will do without making changes:

$ sudo bash ./deploy/install-watchdog.sh --dry-run

Fill in the environment file

The installer copies a template to /etc/ai-bob-setup-agent/watchdog.env. Open it and fill in your API keys: 

$ sudo nano /etc/ai-bob-setup-agent/watchdog.env

Required keys:

ORGO_API_KEY=your-orgo-key
OPENAI_API_KEY=your-openai-key
TELEGRAM_BOT_TOKEN=your-bot-token
TELEGRAM_CONTROL_CHAT_ID=your-chat-id

Optional SMTP settings for email alerts are in the same file — uncomment and configure if you want email in addition to Telegram.

File permissions. The env file is created with chmod 600 so only root can read it. Don't loosen these permissions — API keys are sensitive.

Start and verify

$ sudo systemctl restart ai-bob-watchdog
$ systemctl status ai-bob-watchdog

You should see active (running). Check initial output:

$ journalctl -u ai-bob-watchdog --since "1 minute ago"

Management commands

The Makefile has convenience targets, or use systemctl directly: 

# Check status
$ make watchdog-status
$ systemctl status ai-bob-watchdog

# Tail live logs
$ make watchdog-logs
$ journalctl -u ai-bob-watchdog -f

# View today's logs
$ journalctl -u ai-bob-watchdog --since today

# Restart after config change
$ sudo systemctl restart ai-bob-watchdog

# Stop
$ sudo systemctl stop ai-bob-watchdog

What the service does

The systemd unit is hardened out of the box:

  • Auto-restart on failure — restarts after 10 seconds, up to 5 times in 5 minutes
  • Memory cap — 256 MB limit catches leaks early
  • CPU quota — capped at 25 % of one core
  • Security hardening — NoNewPrivileges, ProtectSystem=strict, PrivateTmp
  • Journald logging — all output goes to the journal, no log rotation needed

Uninstall

To remove the service but keep your env file for a later reinstall:

$ sudo bash ./deploy/uninstall-watchdog.sh --keep-env

To remove everything:

$ sudo bash ./deploy/uninstall-watchdog.sh

Neither command touches the repo, .venv, or customer configs — only the systemd unit and env file.

Troubleshooting

"Python 3.9 found, need >= 3.10"

macOS often ships 3.9. Install a newer Python via brew install python@3.12 and re-run make install. The script will pick up the newer Python automatically.

"Cannot find /agent/workspace"

That path is specific to this repo's CI sandbox. Ignore it.

"OrgoError: 401 unauthorized"

ORGO_API_KEY is either missing or wrong. Double-check you copied it from the Orgo dashboard without trailing whitespace.

"Cannot resolve api.orgo.ai"

DNS issue. Try curl https://api.orgo.ai/health. If that fails too, you have a network problem upstream of the tool.

"ModuleNotFoundError: No module named 'pydantic'"

The venv didn't activate or the install failed. Recreate it: 

$ rm -rf .venv
$ make install

"CI fails on PR but local works"

Most likely the customer YAML you're testing against is in your local config/customers/ (gitignored). CI copies the example YAML automatically; if your test path expects a different slug, adjust the workflow or test.

"ai-bob-watchdog: Main process exited, code=exited, status=1"

The watchdog script crashed on startup. Check the journal for the Python traceback: 

$ journalctl -u ai-bob-watchdog -n 50 --no-pager

Common causes: missing keys in the env file, .venv not set up (run make install), or a syntax error in a customer YAML file.

"Start request repeated too quickly"

The service hit the restart limit (5 restarts in 5 minutes). Fix the underlying error first, then reset the counter: 

$ sudo systemctl reset-failed ai-bob-watchdog
$ sudo systemctl restart ai-bob-watchdog
